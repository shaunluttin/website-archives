Thoughts

Python is a fun language. Through it, I am learning computational theory.

Python is a fun language compared to both C# and JavaScript. It’s similar to JavaScript in its dynamic typing. I wonder how it is going to be different than JavaScript, because Python is moderately typed, whereas JavaScript is weakly typed. Python differs from both C# and JavaScript. Indentations are important for blocks. Further, Python doesn’t require parentheses around conditionals in a while or if block; we just write the expression and then a colon.  Python seems less fascist than C# seems and more so than JavaScript seems. This is a nice balance: I often feel too loose with JavaScript and too tight with C#.

Lecture 1

    • arithmetic  logic unit

    • outputs
  • declarative vs imperative knowledge (statements of fact / tests of truth vs recipes for how to get there)
  • language characteristics
    • high-level vs low-level (how close are you to the machine code?)
    • targeted vs general (is the language for specific types of tasks, e.g. MatLab)
    • interpreted vs compiled (does the language compile to object code before the interpreter sees it?)
  • primitives
    • the very basic computer instructions
    • six basic primitives can accomplish everything
  • program correctness
    • syntax (what is legal)
    • static semantics (light meaning)
    • full semantics and style (full meaning – no help from the compiler or tooling)
  • primitive data elements
    • numbers (int, float)
    • strings
  • expressions vs statements
  • expression
    • some operators working on some operands
    • evaluates to a value
    • yields a value: 2 + 2
    • has no side effects
    • gives the interpreter a value
    • involves operators and operands
1 + 2; // bad, expression trying to be a statement

  • statement/command
    • does something
    • has a side-effect
    • takes a value and asks the computer to do something with it

Lecture 4

  • topics: decomposition, abstraction, recursion
  • review
    • the following programming concepts are Turing complete
    • assignment, conditionals, input/output, looping, data structures
  • functions
    • suppress details (abstraction)
    • create modules and new primitives (decomposition)
  • decomposition
    • add structure to programs with modules
    • isolate components
    • break a program into smaller bits
  • abstraction
    • suppress details
    • hide implementation
    • create a “black box” that has a predictable input and output
  • iterative vs recursive algorithms
    • choosing means knowing which problems
    • naturally are iterative and naturally are recursive
  • function style
    • return something at the end of every branch
    • multiple returns are fine
    • add a specification via comments (constraints, description, inputs, expected return)
  • algorithms – one type is called brute force, e.g. exhaustive search / enumerate and check
  • recursion
    • define the base case, which is the simplest possible solution
    • define the inductive step (aka recursive step),  which breaks the problem into a simpler version of itself, and does some other steps
    • the recursion ends once we reach the base case; we always reach the base case, because the inductive step leads inexorably toward it
    • note: if there are (n) sub problems, then we will need (n) base cases.
    • example: http://repl.it/WbV/8 find a palindrome
  • some python specifics
    • def
    • return
    • None
    • functions have local bindings – the create new scopes / environments / frames
    • functions can nest

Lecture 5

  • integers
    • arbitrary precision
    • long (over 2 billion)
    • long / long = long
  • floats
    • floating point
    • real numbers
    • mantissa (aka significand) and exponent
    • float precision is not exact, so when we compare them, we should compare for closeness (epsilon) not for equality
  • guess, check, and improve algorithm.
    • guess = initial guess
    • enumerate a range, e.g. 100 iterations
      • if guess is close enough
      • then return guess
      • else make a better guess
    • return error if no guess is good enough
  • the bisection method of guessing
    • always guess in the middle of the space of possible answers
    • if the guess is to big/small, the split the space of possible answers
    • this successively splits the space of answers
